# 小说筛选工具正确性与本地接口收口设计

## 背景
当前项目的核心扫描、分组、导出、删除、CLI 与本地 Web UI 已基本实现，但在最终收尾前仍存在两类问题：

1. **功能正确性问题**
   - Web UI 清空扫描目录、扩展名、停用词后，后端可能继续沿用旧值。
   - `include_hidden = false` 时，隐藏目录中的内容仍可能被递归扫描。
   - 删除流程存在“已删一部分再报错”的语义不一致。
   - Web 端保存配置没有跟随当前实际加载的配置路径。
   - CLI `scan` 会在扫描成功前持久化配置，导致临时参数变成长期配置。

2. **localhost API 边界问题**
   - `/api/delete` 过度信任前端传入的 `roots` 与 `paths`。
   - `/api/export` 允许前端指定任意输出路径并直接写入。
   - 一些接口与错误信息会直接暴露本地绝对路径。
   - `/api/pick-folder` 可被任意 localhost 调用触发原生目录选择框。

本轮目标不是处理 Windows 构建环境问题，而是在**不新增大功能**的前提下，优先把实现本身的行为与边界收口。

## 目标
- 修复当前已知的功能正确性问题。
- 将删除权限收口到“最近一次扫描结果中的可删除候选文件集合”。
- 保持导出功能可用，但导出文件统一写入服务端控制的安全导出目录，且只允许基于当前服务端保存的扫描结果执行。
- 减少不必要的绝对路径与内部细节暴露。
- 尽量控制改动范围，集中在已有核心文件中完成。

## 非目标
- 不处理 Windows SDK / `kernel32.lib` / Git Bash linker 环境问题。
- 不引入登录、鉴权系统、会话管理或复杂权限框架。
- 不新增数据库、缓存层或长期任务系统。
- 不重做 UI，仅在现有交互上做最小必要调整。

## 方案对比

### 方案 A：一次性修复正确性与关键安全边界（推荐）
一次改完已确认的问题：请求语义、隐藏目录遍历、删除预校验、配置保存语义、localhost 删除/导出边界。

**优点**
- 相关逻辑主要集中在 `src/model.rs`、`src/web.rs`、`src/scanner.rs`、`src/safety.rs`、`src/main.rs`，便于一次性收口。
- 不依赖本机编译环境恢复，可先把实现质量拉齐。
- 避免后续再次修改同一批接口与状态流。

**缺点**
- 改动面相对方案 B/C 更大。
- Web API 请求与状态管理需要小幅调整。

### 方案 B：只修功能正确性，安全边界下一轮处理
本轮仅处理清空字段、隐藏目录、配置保存、删除语义等问题，删除/导出 API 的权限边界留待下一轮。

**优点**
- 变更量更小。
- 更快把功能行为修到可接受水平。

**缺点**
- 删除与导出接口仍保留明显的 localhost 信任问题。
- 后续还要再次进入同一批文件做改动。

### 方案 C：最小热修
仅修两个高优先级问题：Web 清空字段失效与隐藏目录误扫描。

**优点**
- 变更最少、风险最低。

**缺点**
- 不能完成本轮“收尾”目标。
- 删除流程、配置语义与本地 API 边界仍未解决。

**结论**：采用 **方案 A**。

## 设计概览
本轮设计分为四个收口点：

1. **请求语义收口**：用显式可区分的请求字段表达“不改”与“清空”。
2. **扫描行为收口**：阻止进入隐藏目录，确保 `include_hidden` 符合直觉。
3. **删除授权收口**：由服务端基于最近一次扫描结果决定哪些文件可删，而不是相信客户端提交的 roots/paths。
4. **配置与响应收口**：让配置保存语义更一致，减少 API 对本地路径与内部状态的无差别暴露。

## 模块设计

### 1. `src/model.rs`
#### `ScanRequest` 语义调整
将以下字段从 `Vec<String>` 改为 `Option<Vec<String>>`：
- `scan_dirs`
- `extensions`
- `stopwords`

新的语义：
- `None`：不修改现有配置。
- `Some([])`：明确清空该字段。
- `Some([...])`：用新值覆盖。

`ScanRequest::apply_to()` 也随之调整，保证 Web UI 清空表单时不会继续沿用旧扫描目录、旧扩展名或旧停用词。

#### 删除请求模型调整
将 `DeleteRequest` 从“客户端提交 roots + paths”改为只提交：
- `paths: Vec<String>`

删除边界由服务端掌握，不再由客户端提交 roots。

#### 必须新增：最近扫描授权状态模型
`Engine` 必须记录最近一次扫描的可操作集合，至少包括：
- canonical scan roots
- 最近一次扫描的所有成员文件 canonical path 集合
- 最近一次扫描中“允许删除”的候选集合
- 推荐保留文件集合
- 生成时间或简易 scan token

这份状态只由服务端生成与使用，不暴露给前端直接控制；`/api/delete` 的服务端授权边界直接依赖这份状态，因此它不是可选优化，而是本轮设计的必需部分。

### 2. `src/web.rs`
#### `/api/scan`
- 读取当前配置。
- 应用 `ScanRequest` 中的可选覆盖值。
- 在启动扫描前校验：最终 `scan_dirs` 必须非空。
- 若为空，直接返回错误，不启动扫描。

#### `/api/delete`
- 不再读取客户端传来的 roots。
- 从 `Engine` 中读取最近一次扫描授权状态。
- 对提交的每个 path 进行校验：
  - 必须属于最近一次扫描结果。
  - 必须属于允许删除的候选集合。
  - 不能是推荐保留项。
- 全部路径预校验通过后，再调用删除逻辑。
- 返回结构化结果，而不是把服务端内部路径判断细节直接暴露给客户端。

#### `/api/export`
- 不再允许客户端提交任意绝对输出路径。
- 前端只提交导出格式，以及可选的安全文件名/文件名前缀。
- 服务端将导出文件写入应用控制的固定导出目录（如应用数据目录下的 `exports/`）。
- 导出的内容只能来自当前服务端保存的最近一次扫描报告。
- 不接受客户端上传或拼装报告内容。
- API 响应仅返回成功状态、导出文件名与必要提示，不直接返回完整绝对路径。
- 错误响应尽量返回简化信息，避免把大量绝对路径直接透出。

#### `/api/config`
- WebState 中带上当前实际使用的 resolved config path。
- 若应用启动时使用的是默认配置来源，则在进入 WebState 前就解析出默认配置路径并保存。
- 保存配置时写回当前激活的 resolved config path，而不是固定写默认路径或依赖运行时猜测。

#### `/api/pick-folder`
本轮不引入鉴权，但会在 spec 中明确其风险保持“低优先级接受”，因为该接口只触发 UI，不直接泄露目录内容。若后续需要进一步收紧，可在服务端加入一次性 token 或只允许来自本地页面启动后的会话调用。

#### 错误响应
保留面向 UI 的可读提示，但避免把大量 `path.display()` 拼出来直接返回。对于用户只需要知道“失败原因类别”的接口，优先返回简洁描述。

### 3. `src/scanner.rs`
当前逻辑需要从“跳过隐藏项”升级为“禁止遍历隐藏目录”。

设计要求：
- `include_hidden = false` 时：
  - 隐藏文件不处理。
  - 隐藏目录本身不处理。
  - 隐藏目录下的内容也不进入。
- `include_hidden = true` 时：保持现有行为。

实现方向：
- 使用 `WalkDir::filter_entry(...)` 或等效机制，在目录入口阶段就剪枝。
- 如果已有遍历结构不便直接重写，则至少确保遇到隐藏目录时停止继续递归。

### 4. `src/safety.rs`
删除流程从“边删边验证”调整为“全部验证通过后再删除”。

设计要求：
- 先 canonicalize 全部待删文件。
- 统一验证它们是否都属于服务端允许删除集合。
- 验证全部通过后，再逐个调用回收站删除。
- 日志写入失败不能把已经完成的删除伪装成整体失败。

建议区分“内部结果”与“API 响应”：
- 内部结果可保留 `deleted_count`、canonical deleted paths、日志写入状态等详细信息，供日志记录与调试使用。
- API 响应仅返回：
  - `deleted_count`
  - `requested_count`
  - 被拒绝的原因类别或计数
  - `log_write_warning`（可选）
- API 不直接返回 `deleted_paths`、`rejected_paths`、`log_path` 这类可能暴露绝对路径的信息。

本轮不追求事务式回滚，因为回收站删除本身不适合做真正原子回滚；目标是至少保证**不会在校验未完成时先删除部分文件**。

### 5. `src/main.rs`
#### CLI `scan` 持久化语义收口
推荐调整为：
- `scan` 只应用运行时覆盖参数进行本次扫描。
- 不自动写回配置文件。
- `save-config` 永远写入当前显式指定的 config path；若未显式指定，则写入默认配置路径。

这样带来的好处：
- `scan` 与 `save-config` 的职责更清晰。
- `save-config` 的落盘位置是确定性的，不依赖是否存在 file-backed active config path。
- `--dry-run` 不会出现“逻辑上试运行，实际上写配置”的副作用。
- 扫描失败不会把错误路径或临时参数永久写进配置。

如需保守兼容，也可选“仅在扫描成功后保存”，但本设计推荐彻底分离职责。

## 数据流

### 扫描数据流
1. 前端提交 `ScanRequest`。
2. 服务端基于当前配置应用覆盖值。
3. 校验最终 `scan_dirs` 非空。
4. 执行扫描。
5. 生成 `ScanReport`。
6. 从 `ScanReport` 构建最近扫描授权状态：
   - 最近 scan roots
   - 所有结果成员
   - 可删除候选集合
   - 推荐保留集合
7. 将报告与授权状态一起保存到 `Engine`。

### 删除数据流
1. 前端仅提交勾选的 `paths`。
2. 服务端读取最近扫描授权状态。
3. 对全部 path 做 canonicalize 与授权校验。
4. 若全部通过，再执行回收站删除。
5. 写删除日志（best effort）。
6. 返回结构化结果给前端。

### 导出数据流
1. 前端提交导出格式，以及可选的安全文件名/文件名前缀。
2. 服务端读取当前保存的最近扫描报告。
3. 服务端在固定安全导出目录中生成目标文件名、推断扩展名并写出。
4. 返回简化成功/失败消息与导出文件名。

## 错误处理
- **扫描前校验失败**：直接返回“请先设置扫描目录”等用户可理解消息。
- **删除预校验失败**：返回“所选文件不属于当前扫描结果或不可删除”，不暴露过多内部路径细节。
- **日志写入失败**：删除若已成功，应返回成功结果并附带日志写入警告，而不是整体报错。
- **配置保存失败**：保留可读错误，但避免将完整内部路径堆栈直接回传到前端。
- **导出失败**：前端收到明确失败类型，但服务端内部可保留更详细路径上下文用于诊断。

## 兼容性与迁移影响
- Web 前端需要配合调整请求结构：
  - `ScanRequest` 字段从必传数组改为可选字段语义。
  - `DeleteRequest` 不再传 roots。
- 现有 CLI 命令形式基本不需要变，但 `scan` 的持久化副作用会被移除。
- 已存在默认配置文件格式不需要大规模迁移，因为变更主要发生在 API 请求层与运行时逻辑层。

## 测试策略
即使当前本机构建环境未恢复，本轮设计仍要求补以下测试点：

1. **请求覆盖语义**
   - `None` 不覆盖。
   - `Some([])` 明确清空。
   - `Some([..])` 明确替换。

2. **隐藏目录遍历**
   - `include_hidden = false` 时，不进入隐藏目录。
   - `include_hidden = true` 时，允许进入。

3. **删除授权**
   - 非最近扫描结果文件不可删。
   - 推荐保留文件不可删。
   - 候选文件可删。
   - 删除前若任一文件非法，则整体不开始删除。

4. **配置路径**
   - Web 保存配置会写回当前激活的 config path。

5. **CLI 持久化语义**
   - `scan` 不再写配置。
   - `save-config` 仍负责持久化。
   - `save-config` 未显式指定路径时写默认配置路径；显式指定路径时写该路径。

6. **导出接口**
   - 无当前报告时导出失败。
   - 有当前报告时可导出。

## 风险与取舍
- **不引入鉴权**：因为这是本地单机工具，本轮只收口最关键的服务端授权边界，不引入 session/token 复杂度。
- **导出改为固定安全目录**：牺牲了一部分“任意指定导出位置”的灵活性，换取更明确的 localhost 写入边界与更低的误写风险。
- **删除改为服务端判定**：会让前端协议更严格，但这是本轮最关键的安全收益。
- **不处理构建环境**：短期内无法用完整自动化验证兜底，因此本轮需要靠单元测试设计与源代码审查尽量降低回归风险。

## 实施顺序建议
1. 先改 `model.rs` 请求模型与 `web.rs` 请求解析逻辑。
2. 再改 `scanner.rs` 隐藏目录遍历。
3. 再改 `engine.rs` / `web.rs`，加入最近扫描授权状态。
4. 改 `safety.rs` 删除前全量预校验与结构化结果。
5. 改 `main.rs` 配置持久化语义。
6. 最后补测试并更新 handoff。

## 验收标准
当满足以下条件时，本轮可认为设计目标完成：
- Web UI 清空扫描字段后不会继续沿用旧值。
- `include_hidden = false` 时不会扫描隐藏目录内部内容。
- 删除操作只允许针对最近一次扫描结果中的可删除候选文件。
- 删除在开始执行前已完成全部路径校验。
- Web 保存配置写回当前激活配置路径。
- CLI `scan` 不再偷偷持久化配置。
- 导出只基于服务端当前报告执行。
- API 响应减少不必要的绝对路径暴露。
